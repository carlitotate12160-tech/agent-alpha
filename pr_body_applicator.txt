## Summary

Extract the `CredentialApplicator` seam from `cred_reuse.run` — Step 1 of service-aware cred-reuse. Pure refactor, behaviour unchanged.

**Problem:** `cred_reuse.run` was HTTP-shaped (form-login embedded directly). Adding a DB transport inside it = two-transport god-tool (Lyndon #10/#4/#8).

**Fix:** Split "WHICH credential to try" (cred_reuse orchestration) from "HOW to apply it" (an applicator). One concept per applicator; selection by service via `select_applicator()`.

## Changes

- **`applicator.py`** (new): `AuthResult` dataclass + `CredentialApplicator` protocol + `select_applicator()` + `HttpFormApplicator` (behaviour-preserving extraction of the proven form-login path)
- **`cred_reuse.py`**: `run()` delegates to applicator seam instead of inline HTTP transport. Removed unused `default_creds` helper imports.
- **`test_credential_applicator_seam.py`** (new): 4 tests pinning the seam contract

## Design notes

- `AuthResult` never carries the raw secret (safe fields only)
- `HttpFormApplicator` declares `service="http"`, `required_auth="ACTIVE_APPROVED"`
- Future `MySqlApplicator` (Step 3, DeepSeek lane) will declare `required_auth="OFFENSIVE_APPROVED"` and register in the same `applicators` list
- Baseline GET is now per-credential (inside `apply()`) instead of once before the loop — identical for the proven single-credential chain
